## 严重级别定义

| 级别 | 含义 | 处理要求 |
|------|------|---------|
| **P0** | 阻断性：不修复无法交付 | 必须修复，阻塞 merge |
| **P1** | 严重：影响核心功能或安全 | 必须修复 |
| **P2** | 一般：非核心问题，影响质量 | 评估后决定是否修复 |
| **P3** | 建议：改进建议，非必须 | 记录为技术债务，可后续处理 |

---

## 审计闭环流程

```
问题登记 → 报告 main → main 决策(修复/忽略/技术债务)
    │
    ▼ (如需修复)
coder 修复 → tester 验证 → auditor 终审复核 → main 裁决闭环
    │                                            │
    └── 3轮迭代上限 ──────────────────────────→ 超限由 main 介入
```

## 流程合规检查（终审附加）

| # | 检查项 | 结果 |
|---|--------|------|
| 1 | coder 实现不确定时是否走了升级路径？ | ⬜ |
| 2 | coder-tester 直接交互是否留痕（todo.md/journey.md）？ | ⬜ |
| 3 | todo.md 各阶段是否有 main 确认标记？ | ⬜ |
| 4 | 编译产物是否等测试全通过 + auditor 终审后再交付？ | ⬜ |
