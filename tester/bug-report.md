## 严重级别定义

| 级别 | 含义 | 示例 |
|------|------|------|
| **P0** | 阻断：核心功能不可用，无法继续测试 | 编译失败、启动崩溃、主流程不通 |
| **P1** | 严重：核心功能异常，影响交付 | 数据错误、安全漏洞、关键场景失败 |
| **P2** | 一般：非核心功能异常 | 边界处理不当、UI 问题、性能不达标 |
| **P3** | 建议：改进建议，不影响功能 | 代码可读性、日志优化、命名建议 |

---

## 迭代规则

1. coder 修复后，tester 在 30 分钟内完成验证
2. 每次迭代更新 todo.md（追加子任务）和 journey.md（记录 bug 描述、根因、修复方案、验证结果）
3. 迭代上限 3 轮。第 1 次 bug 和最终通过时告知 main
4. 同一修复引入的新 bug 不计新轮次；独立新 bug 算新轮次
5. 超限后 main 介入：停下分析根因 / 降低要求 / 变更需求

## 测试完成标准

- [ ] 所有可测试项均已覆盖

- [ ] P2 缺陷有明确处理结论（修复/记录技术债务）

- [ ] 测试报告已写入 `docs/test-report.md`
- [ ] journey.md 已记录测试阶段摘要
