# Loop Engineering 模式（实战验证）

> Version: v1.0 | Effective Date: 2026-06-24
> Source: LLM 驱动覆盖率测试项目 45 轮迭代（0% → 100% 分支覆盖）实战验证
> Reference: Addy Osmani Loop Engineering 框架（Goal / Generate / Execute / Verify / Decide）

---

Loop Engineering 是一种"在正确的时间引入正确的人为判断"的自动化循环框架。本文档记录经实战验证的可复用构建块、模式和成本模型。

---

## 一、五大构建块

| 构建块 | 通用做法 | 验证效果 |
|--------|---------|---------|
| **Goal**（目标） | 量化为可自动检测的指标，设停损条件（轮次上限、停滞阈值、预算封顶） | 100% 分支覆盖 |
| **Generate**（生成） | LLM 生成候选方案，注入精确上下文（API 签名、完整条件块、返回模式检测） | 通过率 0% → 100% |
| **Execute**（执行） | 确定性工具自动化（覆盖率工具 + 测试框架 + 隔离沙箱）。先验证后写盘。 | 稳定可靠 |
| **Verify**（验证） | 用指标变化替代 True/False（覆盖率 delta 而非 returncode） | 杜绝假阳性 |
| **Decide**（决策） | 多维度阈值（停滞超过 N 轮、连续无产出、优先级排序） | 8 次关键策略切换 |

### 各构建块详解

#### Goal — 目标可度量、有停损条件

- 目标必须可机器自动检测（如"100% 分支覆盖"而非"测试质量良好"）
- 必须设硬上限：轮次上限、预算封顶、连续停滞阈值
- 分层目标优于单一目标：先突破天花板，再逐模块清零

#### Generate — 上下文决定生成质量

- LLM 生成质量由上下文质量决定，非模型能力
- 必须提供：API 签名清单、导入路径、完整条件块、返回模式检测
- 无需 LLM 猜测返回值——调用成功即覆盖。覆盖后如需验证正确性，再补精确断言
- Few-Shot 注入在覆盖率工具已精确告知缺什么时可能是冗余

#### Execute — 确定性工具，先验证后写盘

- 执行链必须可靠、可重复、有隔离
- **所有文件写入操作必须先隔离验证，通过后再写盘**——这是不可动摇的顺序
- 使用业界成熟工具（Coverage.py / pytest），不自研执行引擎

#### Verify — 用指标变化替代 True/False

- 不信任 returncode——信任覆盖率变化（coverage after - coverage before > 0）
- 停滞检测：连续 N 轮覆盖率无变化 → 触发根因分析而非继续循环
- 回归保护：累积测试保证覆盖率只涨不跌

#### Decide — 多维度信号驱动

- 优先级排序：按文件覆盖率从低到高，0-branch 文件后置
- 停滞终止：连续 N 轮无提升 → 终止本次循环
- 策略切换：同一方案 N 轮无效 → 强制切换策略
- 过滤优于修复：不可覆盖模块在 Generate 阶段直接过滤，不在 Execute 阶段反复尝试

## 二、可复用模式

### 模式 1：过滤优于修复

**场景**：某个模块因环境依赖或架构限制无法被测试覆盖。

**做法**：在 Generate 阶段将该模块从上下文移除——不让 LLM 看到它、不让测试框架尝试执行它。消除所有关联的 token 消耗和时间浪费。

**为什么有效**：不可覆盖模块的反复尝试是最大的迭代轮次黑洞。一个 100 行的不可覆盖模块可能消耗 10+ 轮次仍无产出。彻底过滤将成本降至零。

**适用条件**：确定模块在当前环境下无法覆盖，且过滤不影响核心功能验证。

---

### 模式 2：真文件优于 mock

**场景**：被测函数包含文件系统操作（文件读写、目录创建、路径检查）。

**做法**：用 `tempfile.mkdtemp()` 创建真实临时目录和文件，而非 `unittest.mock.patch()` 模拟文件系统。

**为什么有效**：
- 真实环境更接近生产，测试结论更可信
- LLM 生成真实文件操作代码的成功率远高于生成多层 mock 代码
- 标准库操作，LLM 无需额外理解 mock 框架的复杂语义

**例外**：数据库、网络 API、消息队列等外部服务——必须 mock，但 mock 层数 ≤ 1。

---

### 模式 3：人找 pattern，LLM 实现

**场景**：自动循环在覆盖率深水区（60%+）逐渐失效——剩余未覆盖分支极其复杂或极其细碎。

**做法**：
```
Auditor 分析未覆盖代码的可测性
    ↓
Reviewer 找出可测性最高的函数 + 给出精确测试方案（fixture 构造、触发方式、断言模式）
    ↓
LLM 按方案生成测试代码
    ↓
逐文件清零
```

**为什么有效**：
- LLM 缺乏"选什么目标、用什么策略"的领域判断力
- 人的判断弥补 LLM 的策略盲区
- LLM 的执行能力弥补人的效率瓶颈
- 两者结合优于任何单一手段

**适用条件**：剩余未覆盖分支复杂度高，LLM 自动循环连续 3+ 轮停滞。

---

### 模式 4：累积制

**场景**：多轮迭代的自动化测试生成。

**做法**：
```
通过测试永久保留 → 每轮清覆盖率重跑全部 → LLM 只补新缺口 → 覆盖率只涨不跌
```

**为什么有效**：
- 覆盖率来源从"纯 LLM 本轮生成"变为"累积测试库"
- LLM 负担从"每轮从零开始"变为"只补新缺口"
- 收敛性从"不确定（可能倒退）"变为"自然收敛（只涨不跌）"
- 代码库本身作为状态载体——比 JSON 状态文件更稳固

**对比清场方案**：清场方案中每轮重新生成全部测试，覆盖率会因 LLM 输出的随机性出现倒退。

---

### 模式 5：前置审计

**场景**：设计完成后、编码开始前。

**做法**：Auditor 在设计阶段审查：
1. 需求完整性和无歧义性
2. 设计方案是否与业界共识一致（如不一致，差异的理由是否充分）
3. 宪章一致性检查
4. 安全风险识别
5. 测试可行性评估

**为什么有效**：设计阶段发现问题的成本是编码阶段的 1/5。设计审查拦截的是方法层面的错误，一旦进入编码，修复成本呈指数增长。

**适用条件**：M 级及以上项目必须执行。S 级项目可用 main 自检代替。

## 三、成本经验

以下为通用成本参考，基于 LLM 驱动覆盖率测试实战数据：

| 项目类型 | 预计成本（参考） | 预计时间 | 建议预算 |
|---------|--------------|---------|---------|
| 小项目（< 10 文件） | 低 | 1-2 小时 | 实际消耗的 5 倍 |
| 中项目（10-50 文件） | 中 | 3-5 小时 | 实际消耗的 5 倍 |
| 大项目（50+ 文件） | 高 | 5-10+ 小时 | 实际消耗的 10 倍 |

### 成本控制原则

1. **预算为实际消耗的 10 倍可保证不中断。** 预算过紧会在临近完成时因余额不足中断——最浪费的无谓损耗。
2. **每轮后查询余额。** 余额低于 20% 时暂停循环，评估是否继续。
3. **计价模型应使用实际模型定价，而非过时参考。** 不同模型之间定价可差 10 倍。
4. **Token 效率评估。** 每次 prompt 增强必须评估 token 增量 vs 预期覆盖率增量。砍掉无增益的内容。

## 四、反模式（必须避免）

| # | 反模式 | 后果 | 替代做法 |
|---|--------|------|---------|
| 1 | 为 LLM 设计超过 2 层抽象的接口 | LLM 无法正确使用，连续多轮 0% 通过率 | 优先真实环境，mock 层数 ≤ 1 |
| 2 | 先写文件后验证 | 孤儿文件污染、覆盖率归零 | 先隔离验证语法和兼容性，通过后再写盘 |
| 3 | 同一方案死磕超过 3 轮 | 浪费 token 和轮次，无产出 | 连续 3 轮无效 → 强制切换策略 |
| 4 | 仅凭 returncode 判断有效性 | 假阳性——崩溃被报告为通过 | 用覆盖率变化替代 returncode |
| 5 | Pure LLM 自动循环在深水区硬撑 | 反复停滞，无法突破 | 60%+ 切换到 human-in-the-loop |

## 五、Human-in-the-Loop 触发条件

以下信号出现时应引入人工（Auditor / Reviewer）判断：

| 信号 | 动作 |
|------|------|
| 连续 3 轮覆盖率无提升 | 暂停自动循环，Auditor 分析未覆盖分支可测性 |
| 同一模块连续 2 轮无新增覆盖 | 评估是否应过滤该模块 |
| 覆盖率超过 60% 后增速显著放缓 | 切换到"人找 pattern，LLM 实现"模式 |
| 异常路径（try/except）大面积未覆盖 | Reviewer 逐函数给出异常触发方案 |
| 测试通过率低于 50% | 检查 prompt 是否缺少关键上下文（API 签名、导入路径、返回模式） |

## 六、与 Loop Engineering 框架的对齐

| 维度 | Loop Engineering 理念 | MA 实践 |
|------|----------------------|---------|
| **成本** | 每次循环评估成本，设置硬上限 | 预算封顶 + 每轮余额检查 + 实际模型定价 |
| **状态** | 循环间状态持久化、可追溯 | 累积测试制 + 代码库本身作为状态载体 |
| **失控** | 防止无限循环、方向错误、质量退化 | 连续停滞终止 + 轮次硬上限 + 策略切换阈值 |

### MA 框架的独特贡献

1. **累积制是"状态"实现的典范。** 测试代码库作为状态载体，比 JSON 文件更稳固——git 提供了完整的历史追溯和回滚能力。
2. **"过滤优于修复"是"Decide"智能的体现。** 决定"不让 LLM 看到它"比"反复尝试修复"更高效——这是决策环节的判断力。
3. **"人找 pattern，LLM 实现"是 human-in-the-loop 的最佳实践。** 循环不意味着全自动。在深水区引入人的判断用于"决定下一步"，LLM 用于"生成行动"——各取所长。
4. **"不改源码"是安全基线。** 自动化循环最大的风险是对系统造成不可逆破坏。这条基线在自动化循环启动前就应确立。
